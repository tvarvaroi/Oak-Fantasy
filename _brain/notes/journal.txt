DETOUR — Spirală debug "42501 RLS regression":

Smoke test cu `.from('email_subscribers').insert({email}).select()` ca
anon returna 42501. Am presupus RLS regression după ALTER TABLE RENAME
(migration 20260522090009). Am pushed 5 migrations debug:
- 20260615180000 — "Allow anon subscribe" WITH CHECK true (ghost fix)
- 20260615190000 — drop+recreate "Anyone can subscribe"
- 20260615200000 — alter WITH CHECK true
- 20260615210000 — explicit grants + NOTICEs
- 20260615220000 — NOTIFY pgrst reload

Niciuna n-a rezolvat. Audit detaliat al pg_policies arăta state corect.
Service role INSERT funcţiona. Anon JWT verified pentru proiectul corect.

ROOT CAUSE (descoperit 2026-06-16 după restart proiect + retry):

`supabase-js .insert(entry).select()` cere PostgREST să returneze rândul
inserat printr-un SELECT post-INSERT. Anon NU are SELECT policy pe
email_subscribers — by design, doar service_role citeşte subscribers.
PostgREST mapează "INSERT succeeds but SELECT-back blocked" la cod
42501 misleading. Producţia foloseşte `.insert(entry)` fără chain
`.select()`. Producţia n-a fost niciodată broken.

CLEANUP final (canonical restore):

1. Migration nouă `20260616120000_restore_canonical_email_subscribers_policies.sql`:
   - Idempotent drop "Allow anon subscribe" + "Anyone can subscribe"
   - Recreate "Anyone can subscribe" cu canonical WITH CHECK email format
   - Defense-in-depth alongside Zod app-layer validation

2. `supabase migration repair --status reverted` pe toate 5 ghost migrations
   (180000, 190000, 200000, 210000, 220000). Tracking sincronizat cu disk.

3. Şters ghost fix `20260615180000_restore_anon_insert_email_subscribers.sql`
   din disk local. Repo 100% clean pentru prod fresh deploy.

4. Smoke test corect (scripts/smoke-canonical.mjs):
   - T1: production path `.insert()` fără `.select()` → PASS 201
   - T2: invalid email → 42501 (WITH CHECK enforcing) → PASS
